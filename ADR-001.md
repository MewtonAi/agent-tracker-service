# ADR-001: API-First Agent Tracker Service using Micronaut + Mongo Atlas + MCP

- **Status:** Accepted
- **Date:** 2026-02-23
- **Deciders:** agent-tracker-service maintainers
- **Related docs:** `ARCHITECTURE.md`

## Context

We need a task-tracking backend for agent orchestration that is:
- Fast to ship and simple to operate
- Usable by traditional services and AI agents
- Capable of storing flexible task metadata with strong lifecycle semantics
- Ready for cloud operation with minimal platform overhead

The existing scaffold already includes Micronaut, MongoDB data support, and Micronaut MCP server dependencies.

## Decision

We will implement a **modular monolith** in **Java 21 + Micronaut** with:
1. **API-first contracts**
   - REST (`/v1`) for service integrations
   - MCP tools/resources for agent integrations
2. **Mongo Atlas** as primary persistence
   - `tasks` collection for canonical state
   - Optional `task_events` append-only collection in phase 2
3. **Shared application services** for business logic
   - REST and MCP adapters both call the same use-case layer
4. **Strict validation + normalized errors**
   - Bean validation at boundary
   - domain transition rules for state changes
   - consistent REST and MCP error mapping
5. **Idempotent mutations + optimistic locking**
   - idempotency keys / external IDs
   - version field for concurrency safety

## Why this decision

- **Micronaut**: low overhead, fast startup, strong compile-time tooling, native-friendly.
- **Mongo Atlas**: flexible document model fits variable task metadata and quick iteration; managed service reduces ops burden.
- **MCP + REST dual exposure**: allows both deterministic service integration and AI-native tool usage without duplicating business logic.
- **Modular monolith first**: faster delivery and simpler debugging than microservices at current scope.

## Alternatives considered

### 1) Relational DB (PostgreSQL) + JPA
- **Pros:** strong relational guarantees, SQL analytics
- **Cons:** higher schema migration friction for rapidly evolving task metadata
- **Result:** deferred; may be revisited if relational reporting dominates

### 2) MCP-only interface (no REST)
- **Pros:** optimized for agent consumers
- **Cons:** poor fit for dashboards/backends and non-agent systems
- **Result:** rejected; REST remains necessary

### 3) Microservices split from day 1
- **Pros:** strict bounded contexts
- **Cons:** operational overhead, slower MVP, more failure modes
- **Result:** rejected for initial phase

## Consequences

### Positive
- One deployable with clear internal boundaries
- Consistent behavior across REST and MCP
- Good developer velocity for phase 1/2
- Cloud-ready persistence and scaling path via Atlas

### Negative / risks
- Document model can drift without schema discipline
- Query complexity and indexing need active tuning
- Single service boundary may eventually require decomposition

## Guardrails

- Maintain explicit DTO schemas and avoid unchecked metadata growth
- Enforce index policy for all production query paths
- Keep transition rules in domain layer only (single source of truth)
- Add integration tests for REST/MCP parity

## Implementation notes (phase 1)

1. Define canonical task model + status transition matrix
2. Implement `TaskCommandService` / `TaskQueryService`
3. Add REST endpoints:
   - `POST /v1/tasks`
   - `GET /v1/tasks/{id}`
   - `GET /v1/tasks`
   - `PATCH /v1/tasks/{id}/status`
4. Add MCP tools:
   - `create_task`, `get_task`, `list_tasks`, `update_task_status`
5. Add global error mapping and correlation IDs
6. Configure Atlas indexes and optimistic locking

## Integration guidance for other agents/services

### Create task
- Prefer REST for service calls; MCP for agent-native calls
- Must provide either:
  - `externalTaskId`, or
  - idempotency key for mutation
- Include `sourceSystem` and `requester`

### Read task(s)
- `GET /v1/tasks/{id}` or MCP `get_task`
- Use filtered list APIs with pagination/cursors; avoid unbounded scans

### Update task
- Use dedicated transition endpoint/tool (`update_task_status`)
- Handle `409 CONFLICT` by reloading current task and retrying intentfully

## Revisit triggers

Re-evaluate this ADR if:
- Task volume/query complexity exceeds Atlas/indexing comfort zone
- Need for strict multi-entity ACID workflows increases
- Distinct ownership domains justify service decomposition
- Security/tenancy requirements require stronger isolation boundaries
